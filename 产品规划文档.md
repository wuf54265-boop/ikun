## 0. 项目定位与价值主张

### 0.1 一句话定位
面向电商 / 零售业务人员的 **「可解释 AI 数据分析助手」**：上传一份表格，即可获得统计建模结果 + 自然语言洞察 + 可视化报告；核心统计方法透明、可查、可复现，AI 只做「翻译」不做「黑盒结论」。

### 0.2 产品层目标用户（Persona）
| 角色 | 场景 | 核心诉求 |
|------|------|----------|
| 电商 / 零售运营分析师 | 周报、活动复盘、用户分层 | 快速出图出结论，少写代码 |
| 中小企业数据分析师 / 业务 BP | 没有现成 BI，临时取数分析 | 一键完成清洗+检验+建模 |
| 业务管理者 | 看不懂 Python，但要看结论 | 自然语言 + 可视化，结论可追溯 |

### 0.3 价值主张（差异化壁垒）
1. **统计硬实力可见**：回归、聚类、假设检验、分布检验等核心算法**自实现**，结果附带公式与假设说明。
2. **行业模板内置**：RFM 用户分层、转化漏斗等电商高频分析「开箱即用」。
3. **全过程可解释**：每个结论标注「用了什么方法 / 为什么用 / 假设是什么 / 局限在哪」。
4. **AI 不黑盒**：LLM 仅把结构化计算结果翻译成人话；所有数字来自真实统计计算，绝不凭空生成。

### 0.4 对求职的三重支撑
- **统计建模能力**：自实现 OLS 回归、K-Means、假设检验、分布检验、相关性分析等 **8 项核心算法**（Isolation Forest 异常检测调用 sklearn，取舍见第 6 节）。
- **AI 应用能力**：LLM 编排、结构化结果→自然语言 prompt 工程、可控输出。
- **产品开发思维**：PRD、架构、数据流、页面结构、里程碑、优先级取舍（见下文）。

## 1. PRD 产品需求文档

### 1.1 用户角色（系统层）
- **终端用户（分析者）**：上传数据、查看报告、调整参数。
- **访客**：未登录也能试用公共 Demo 数据集。
- **系统 / 面试官（隐式）**：通过项目本身评估作者工程与统计能力。

### 1.2 功能模块与用户故事（含验收标准 AC）

#### Epic A — 数据接入
- **US-A1 上传文件**：作为分析者，我想上传 CSV / Excel，系统自动解析为可分析的数据集。
  - AC1：支持 `.csv` / `.xlsx` / `.xls`；单文件 ≤ 50MB；编码自动探测（UTF-8 / GBK）。
  - AC2：解析后展示行列数、字段列表、前 20 行预览。
  - AC3：解析失败给出明确错误原因（如编码错误、空文件）。

#### Epic B — 自动数据理解（Profiling）
- **US-B1 字段类型识别**：系统自动推断数值 / 类别 / 日期 / 文本 / 布尔，并允许手动校正。
  - AC：对每列给出推断类型 + 置信度 + 样例值。
- **US-B2 描述统计**：对数值列给出均值、中位数、标准差、分位数、偏度、峰度；对类别列给出取值分布、Top 值。
  - AC：结果以表格 + 迷你分布图呈现。
- **US-B3 数据质量报告**：缺失率、重复行、常量列、潜在异常格式、高基数 ID 提示。
  - AC：输出质量评分（0–100）与问题清单。

#### Epic C — 数据清洗
- **US-C1 缺失值处理**：检测每列缺失比例，给出处理建议（删除 / 均值 / 中位数 / 众数 / 模型填补），用户可一键应用或自定义。
  - AC：应用后可预览清洗前后差异。
- **US-C2 异常值检测**：提供 IQR、Z-score、Isolation Forest 三种方法（IQR / Z-score 自实现，Isolation Forest 调用 sklearn），标注异常点并可视化。
  - AC：用户可切换方法、调整阈值；输出异常比例与索引。

#### Epic D — 统计分析
- **US-D1 相关性分析**：数值列两两 Pearson / Spearman 相关 + 显著性，输出热力图。
  - AC：可切换相关系数类型；标注显著性水平（* / **）。
- **US-D2 假设检验**：两样本 t 检验（Welch）、比例 Z 检验、卡方独立性检验，给出统计量、p 值、结论。
  - AC：用户输入分组列与指标列即可运行；输出原假设 / 结论 / 效应量。
- **US-D3 分布检验**：正态性检验（KS / Shapiro-Wilk）、拟合优度，判断数据分布形态。
  - AC：输出检验统计量、p 值、是否拒绝正态性假设。

#### Epic E — 建模分析
- **US-E1 回归分析**：OLS 线性回归（自实现），输出系数、标准误、t 值、p 值、R²、调整后 R²、残差诊断图。
  - AC：支持多变量；可勾选标准化系数。
- **US-E2 聚类分析**：K-Means（自实现）+ 肘部法 / 轮廓系数选 K；输出簇标签、质心、聚类散点图。
  - AC：用户指定特征列与 K（或自动选 K）；高维用 PCA 降维后可视化。

#### Epic F — 行业模板（差异化）
- **US-F1 RFM 分析**：基于交易数据自动计算 R/F/M，五分位打分，用户分群（冠军 / 忠诚 / 流失风险等），给出运营建议。
  - AC：需用户映射「客户ID / 日期 / 金额」三列；输出分群占比 + 人群画像。
- **US-F2 漏斗分析**：基于事件流数据计算各步骤转化率与流失点。
  - AC：用户按顺序选择步骤列；输出漏斗图 + 各环节转化率。

#### Epic G — AI 分析报告
- **US-G1 自然语言洞察**：系统汇总上述所有结构化结果，调用 LLM 生成中文分析报告（摘要 + 关键发现 + 建议 + 风险）。
  - AC：报告中的数字必须来自真实计算结果（结构化注入，非 LLM 自由发挥）；支持「重新生成 / 调整语气」。
- **US-G2 可解释输出**：每个分析卡片显示「方法说明 + 假设 + 解释」，点击可展开公式与推导。
  - AC：无黑盒结论；AI 段落可溯源到具体分析模块。

#### Epic H — 可视化与导出
- **US-H1 自动图表**：相关性热力图、分布直方图 / 箱线图、聚类散点图、RFM 矩阵、漏斗图。
  - AC：图表由后端返回 spec，前端用 Recharts 渲染。
- **US-H2 报告导出**：导出 Markdown / PDF 版分析报告（P2）。

### 1.3 功能优先级矩阵

| 优先级 | 功能 | 理由 |
|--------|------|------|
| **P0（MVP 必须）** | 文件上传(CSV/Excel)、字段类型识别、描述统计、数据质量报告、缺失值检测与处理、相关性分析(Pearson)、OLS 回归(自实现)、基础可视化(分布图/热力图)、AI 报告生成、可解释输出 | 缺任一则无法演示「完整分析闭环」，且无法体现统计+AI+产品三重能力 |
| **P1（核心差异化）** | 异常值检测(IQR/Z-score 自实现 + Isolation Forest 调用 sklearn)、假设检验(Welch t/卡方)、分布检验、K-Means 聚类(自实现)、RFM 模板、漏斗模板 | 体现「自实现算法」与「电商行业聚焦」，是区别于 AI Wrapper 的关键 |
| **P2（增强与打磨）** | 多模型对比、PCA 降维可视化、报告 PDF 导出、历史会话、用户账户、自定义算法参数、示例数据集市场、暗色模式 | 提升完成度与可用性，非面试必需 |

### 1.4 非功能需求（NFR）
- **可解释性**：所有结论可溯源到计算与假设（硬性要求）。
- **隐私**：数据默认仅会话内处理，不上云持久化；明确告知用户。
- **性能**：50MB / 10 万行内，单步分析 < 3s（P0）；超大数据给出采样提示。
- **可移植**：核心算法库 `stats_lib` 与框架解耦，可独立单元测试与复用。
- **存储**：数据集落盘存储（Parquet + SQLite 元数据），不使用进程内内存字典（见 2.6）。

### 2.1 系统架构图（文字 / ASCII 描述）

要点：前端只负责「呈现与交互」；后端把分析拆成独立服务；**stats_lib 自实现算法库与 FastAPI 解耦**，可单独测试与展示，这正是面试亮点；数据集通过存储层落盘，而非驻留内存。

### 2.2 模块划分
**前端（Next.js App Router）**
- `app/`：各页面路由（见第 4 节）
- `components/`：上传组件、图表封装（Recharts）、分析卡片、可解释面板
- `lib/api.ts`：封装后端调用 + 统一响应解析
- `store/`：会话级数据集状态（**Zustand**）—— 见下方选型说明

**后端（FastAPI）**
- `routers/`：`datasets / profiling / cleaning / stats / modeling / templates / insight / viz`
- `services/`：上述各服务，编排 stats_lib 与 pandas
- `core/stats_lib/`：**自实现算法**（第 6 节），纯函数，输入 DataFrame/ndarray，输出结构化结果 + explanation
- `schemas/`：Pydantic 请求/响应模型，含 `explanation` 字段
- `ai/`：LLM 客户端 + prompt 模板（结构化结果 → 报告）
- `store/`：数据集存储层（`data/{dataset_id}.parquet` 落盘）+ SQLite（会话/元数据）。**不用内存字典**，避免重启丢数据、多用户并发吃光内存。

**前端状态管理选型：Zustand（已定，不再留待下一步）**
- 项目跨多个分析模块共享状态：当前数据集、各步分析结果、图表配置、向导进度。
- 不选 React Context：Context 任一值变化会触发所有消费者重渲染，分析模块多、状态大时会明显卡顿，且需大量 Provider 嵌套。
- 选 Zustand 的理由：① 选择器（selector）精准订阅，只有用到某片的组件才重渲染；② 样板代码极少、学习成本低，适合个人项目；③ 可在组件外（如 API 回调）直接读写 store，便于编排多步分析；④ 自带 `persist` 中间件，可把轻量状态存 localStorage 防刷新丢失（**注意：只持久化轻量元数据，原始 DataFrame 始终在后端存储层，不进前端状态**）。
- 落地：阶段一搭骨架时直接引入 Zustand，建立 `useDatasetStore` / `useAnalysisStore` 两个 slice。

### 2.3 技术选型与理由
| 层 | 选型 | 理由 |
|----|------|------|
| 前端 | Next.js + Tailwind + Recharts | 就业市场主流；SSR/部署简单；Recharts 与 React 契合 |
| 状态管理 | **Zustand** | 精准订阅、样板少、适合多模块共享状态（见 2.2） |
| 后端 | FastAPI | 异步、自动 OpenAPI、Pydantic 校验，适合数据 API |
| 数据处理 | Pandas + NumPy + SciPy | 行业标准；SciPy 用于分布 CDF/p-value 等底层数学 |
| 自实现 | 纯 NumPy（核心算法手写；Isolation Forest 用 sklearn） | 体现统计功底（面试卖点），把工程量留给高回报算法 |
| 异常检测 | sklearn IsolationForest（仅此一项用库） | 自实现工程量/收益比不划算，取舍见 6.2 |
| AI | OpenAI API | 稳定、可控；结果结构化注入避免幻觉 |
| 存储 | 文件系统 Parquet + SQLite | 零依赖、开发期直接可用；接口隔离，生产可换持久卷/对象存储 |
| 部署 | Vercel（前端）+ Railway（后端） | 免费额度友好，CI 简单，适合作品集 |

### 2.4 API 设计概览（统一响应信封）
```json
{ "data": {...}, "explanation": {...}, "meta": { "method": "...", "assumptions": [...] } }
```
- `POST /api/v1/datasets/upload`
- `GET  /api/v1/datasets/{id}/profile`
- `GET  /api/v1/datasets/{id}/quality`
- `POST /api/v1/datasets/{id}/clean`  （body: 各列策略）
- `POST /api/v1/datasets/{id}/anomalies` （method: iqr|zscore|isolation_forest）
- `POST /api/v1/analysis/correlation`  （type: pearson|spearman）
- `POST /api/v1/analysis/hypothesis`   （test: welch_t|prop_z|chi2）
- `POST /api/v1/analysis/distribution` （test: ks|shapiro）
- `POST /api/v1/modeling/regression`   （target, features, standardize?）
- `POST /api/v1/modeling/clustering`   （features, k?, method: kmeans）
- `POST /api/v1/templates/rfm`         （customer_id, date, amount 映射）
- `POST /api/v1/templates/funnel`      （ordered step columns）
- `POST /api/v1/insight/report`        （汇总前面所有 results）
- `GET  /api/v1/viz/{spec_id}`         （返回 Recharts spec）

### 2.5 可解释性设计（关键）
每个分析返回三件套：`data`（结果）+ `explanation`（方法/假设/解读）+ `meta`（参数）。前端「可解释面板」默认折叠公式，点击展开。AI 报告生成时，prompt 强制「只基于传入的结构化数据陈述，不得编造数字」。

### 2.6 会话存储设计（数据集落盘，非内存字典）
- **选型**：开发阶段用**文件系统 Parquet**，每个数据集按 `dataset_id`（uuid）落盘；**不使用进程内内存字典**，原因：① 进程/容器重启会丢数据；② 多用户并发时每个 50MB DataFrame 驻留内存会迅速吃光 RAM。
- **目录与文件**（后端 `data/` 目录，绝对路径在配置中）：
  - `data/{dataset_id}.parquet`：上传解析后的原始 DataFrame（分析全程的单一事实源）
  - `data/{dataset_id}_clean.parquet`：清洗后版本（保留原始可追溯，不覆盖原文件）
  - `data/{dataset_id}_profile.json`：概览/质量结果缓存（可选，加速二次访问）
- **元数据**：SQLite 表 `datasets(id TEXT PK, filename, rows, cols, size_mb, created_at, status)`，用于数据集列表与生命周期管理；原始大 DataFrame 不进 SQLite（仅存路径与摘要）。
- **并发与隔离**：`dataset_id` 用 uuid 防冲突；写操作采用「写临时文件 → 原子 rename」或文件锁，避免并发写损坏。
- **生命周期（P2）**：会话结束或 N 天未访问可清理；提供 TTL 清理任务，清理前提示用户导出。
- **生产适配**：Railway 文件系统为临时卷，生产应挂**持久卷（Railway Volume）**或换对象存储（S3 / Cloudflare R2）。存储层用接口隔离（`store/dataset_repo.py` 定义抽象方法），切换底层只需改实现，业务/路由层无感。

## 3. 数据流程图（关键数据流文字描述）

### 3.1 主流程（一次完整分析）
1. **接入**：用户上传 CSV/Excel → 后端 `Ingestion` 解析为 DataFrame，推断编码 → 生成 `dataset_id`（uuid）→ 原始 DataFrame 落盘为 `data/{dataset_id}.parquet`，摘要写入 SQLite → 返回预览。
2. **理解**：前端请求 `/profile` 与 `/quality` → `Profiling` 调用 stats_lib 的类型识别 + 描述统计 + 质量评分 → 返回字段字典与质量报告（可选缓存到 `_profile.json`）。
3. **清洗（可选）**：用户查看缺失/异常建议 → 选择策略 → `Cleaning` 生成清洗后 DataFrame，另存为 `data/{dataset_id}_clean.parquet`（保留原数据可追溯）→ 返回前后对比。
4. **统计分析**：用户选择相关性 / 假设检验 / 分布检验 → `Stats` 调用对应自实现函数 → 返回统计量 + p 值 + 解释 + 图表 spec。
5. **建模**：用户配置回归/聚类 → `Modeling` 调用 OLS / K-Means 自实现 → 返回系数/簇标签 + 残差图/聚类图 spec。
6. **行业模板（可选）**：选择 RFM / 漏斗 → `templates` 按映射列计算 → 返回分群/转化率 + 图表。
7. **可视化**：所有 spec 汇总 → 前端 Recharts 渲染。
8. **AI 报告**：前端把上述所有 `data` + `explanation` 打包发给 `/insight/report` → 后端组装 prompt → 调用 OpenAI → 返回中文报告（数字全部来自真实结果）。
9. **呈现**：报告页聚合「可解释分析卡片 + 图表 + AI 叙事」，形成可交付物。

### 3.2 异常流
- **大文件**：超 50MB 或超 10 万行 → 提示采样分析（如随机 5 万行）并标注「基于样本」。
- **解析失败**：编码/格式错误 → 返回明确错误码与原因，引导用户重新导出。
- **LLM 超时/失败**：降级为「结构化结果卡片 + 提示手动撰写」，保证核心分析可用。
- **算法不收敛**（如 K-Means 极端数据）：返回警告 + 建议（标准化 / 换特征）。
- **存储写失败**：落盘异常时返回明确错误，避免脏数据；前端提示重试。

## 6. 需要自己实现的统计方法清单及算法描述

> 约定：**「自实现」= 不调用 sklearn 等现成模型**，核心数值计算用 NumPy 手写；分布 CDF / p 值等底层数学可借用 SciPy（`scipy.stats.t` / `chi2` / `norm`），属于合理边界。所有方法均输出 `explanation`（方法、假设、解读）以满足可解释性。Isolation Forest 是**唯一**调用 sklearn 的算法，取舍见 6.2。

### 6.1 方法总表
| 方法 | 类型 | 自实现? | 用途 |
|------|------|---------|------|
| K-Means 聚类 | 无监督 | ✅ 自实现 | 用户/商品分群 |
| Isolation Forest 异常检测 | 无监督 | 📦 sklearn（不自实现） | 深度异常检测（自实现性价比低，见 6.2） |
| OLS 线性回归 | 监督 | ✅ 自实现 | 影响因素建模 |
| Welch t 检验 | 假设检验 | ✅ 自实现 | 两组均值差异 |
| 卡方独立性检验 | 假设检验 | ✅ 自实现 | 类别变量关联 |
| KS 正态性检验 | 分布检验 | ✅ 自实现 | 数据分布形态 |
| Pearson 相关 + 显著性 | 相关 | ✅ 自实现 | 变量关系 |
| IQR / Z-score 异常 | 异常 | ✅ 自实现（简单） | 快速异常初筛 |
| RFM 引擎 | 行业 | ✅ 自实现 | 用户价值分层 |
| 漏斗转化率 | 行业 | ✅ 自实现 | 步骤转化分析 |
| PCA 降维（可视化用） | 降维 | 可用 sklearn（非核心） | 高维聚类可视化 |

> 实际自实现 **8 项核心统计方法**（Isolation Forest 调用 sklearn），远超「至少 2 项」要求，且覆盖聚类、异常、回归、检验、分布、相关六大类，统计广度与深度兼备。

### 6.2 算法简要描述

#### (1) K-Means 聚类（自实现）
- **初始化**：k-means++（按与已选质心距离平方成比例概率选下一个质心），避免坏初始化。
- **迭代**：① 每个样本分配到最近质心（欧氏距离）；② 质心更新为簇内均值；③ 重复至质心位移 < tol 或达 max_iter。
- **选 K**：计算不同 K 的「惯性 inertia = Σ‖x−μ‖²」画肘部图；并用**轮廓系数**（自实现：a=样本到同簇其他点平均距离，b=到最近其他簇平均距离，s=(b−a)/max(a,b)）辅助。
- **输出**：簇标签、质心、inertia、轮廓系数、各簇规模。

#### (2) Isolation Forest 异常检测（调用 sklearn，附取舍说明）
- **决策：不手写，直接调用 `sklearn.ensemble.IsolationForest`。** 原因（投入产出比）：
  - 自实现需递归建树、路径长度累积、异常分数 `s(x)=2^(−E[h(x)]/c(n))` 及 `c(n)=2H(n−1)−2(n−1)/n`（H 为调和数），工程量约为其余自实现算法的数倍。
  - 对面试加分的边际收益很低——面试官不会因为你会手写 Isolation Forest 而显著加分，但因此拖延进度、挤压部署与技术博客时间的损失很大。
- **做法**：封装在 `services/cleaning` 中，对外接口与 IQR/Z-score 完全一致（输入 DataFrame → 异常索引 + 分数 + 解释），前端无需为它特殊分支。
- **可解释性不降级**：仍输出每个异常点的分数、关键特征与「为何判为异常」的解读，满足非黑盒要求。
- 省下的时间投入到**部署上线**与**技术博客**（见里程碑阶段四），整体求职收益更高。

#### (3) OLS 线性回归（自实现）
- **系数**：`β = (XᵀX)⁻¹ Xᵀy`，用 `np.linalg.solve` 解正规方程（比显式求逆更稳）。
- **拟合**：预测 `ŷ = Xβ`；残差 `e = y − ŷ`；RSS、MSE、R²、调整 R²。
- **推断（自实现）**：`σ² = RSS/(n−p)`；`se(β) = √diag(σ²(XᵀX)⁻¹)`；`t = β/se`；p 值由 t 分布 CDF 得（SciPy 取 CDF 属合理边界）。
- **输出**：系数表（估计/标准误/t/p）、R²、残差诊断图数据。可勾选标准化系数。

#### (4) Welch t 检验（自实现）
- **统计量**：`t = (x̄₁−x̄₂) / √(s₁²/n₁ + s₂²/n₂)`（不假设方差齐性）。
- **自由度**：Welch–Satterthwaite `df = (a+b)² / (a²/(n₁−1) + b²/(n₂−1))`，a=s₁²/n₁, b=s₂²/n₂。
- **p 值**：双尾 `2·(1−CDF_t(|t|,df))`。输出原假设、结论、效应量（Cohen's d）。

#### (5) 卡方独立性检验（自实现）
- 由列联表 O，计算期望 `E = (行和×列和)/N`；`χ² = Σ(O−E)²/E`。
- 自由度 `df=(r−1)(c−1)`；p 值由卡方分布 CDF 得；输出关联强度（Cramér's V）。

#### (6) KS 正态性检验（自实现）
- 经验 CDF `F_n(x)` 与正态参考 CDF `Φ((x−x̄)/s)` 比较；`D = max|F_n − Φ|`。
- p 值用 KS 分布近似（或大样本临界值）；输出是否拒绝正态性假设，指导后续选参（如用非参检验）。

#### (7) Pearson 相关 + 显著性（自实现）
- `r = cov(x,y)/(σ_x σ_y)`；显著性 `t = r·√((n−2)/(1−r²))`，p 由 t 分布得；标注 * (p<.05) / ** (p<.01)。

#### (8) IQR / Z-score 异常（自实现，简单）
- IQR：`异常 if x < Q1−1.5·IQR or x > Q3+1.5·IQR`。
- Z-score：`z=(x−μ)/σ`，`|z|>3` 判异常（可调阈值）。

#### (9) RFM 引擎（自实现，行业）
- 由交易表聚合每个客户的 R（最近购买距今天数）、F（购买次数）、M（总金额）。
- 三分位/五分位打分（1–5）；RFM 组合映射分群标签（冠军 / loyal / 有潜力 / 流失风险 / 已流失等）；输出分群占比与运营建议模板。

### 6.3 验证策略（保证「自实现也正确」）
- 对每个自实现函数写单元测试，与 `scipy` / `sklearn` 在相同数据上对照，容许浮点容差（如 `atol=1e-6`）。
- 示例：K-Means 标签与 sklearn 在 Iris 上**调整兰德指数 ≥ 0.9**；OLS 系数与 `statsmodels` 误差 < 1e-6；Welch t / 卡方 / KS 与 `scipy` 误差在容差内。
- 在 `/methods` 页与 README 展示对照结果，证明自实现既「手写」又「正确」；同时说明 Isolation Forest 直接复用 sklearn 的工程取舍。

## 附：关键技术决策备忘
- **自实现 vs 库**：核心算法手写（面试卖点），分布 CDF/p 值用 SciPy（合理边界）；**Isolation Forest 与 PCA 用 sklearn**（工程量/收益比不划算，取舍见 6.2）；其余 **8 项**自实现。

- **会话存储 = 文件系统 Parquet + SQLite**（已定）：不用内存字典，重启不丢、并发不吃内存；接口隔离，生产可换持久卷/对象存储。
- **AI 边界**：LLM 只做结构化结果的「翻译」，所有数字来自计算，避免幻觉与黑盒。
- **可解释性贯穿**：响应信封强制含 `explanation`，前端可展开公式。
- **解耦**：`stats_lib` 独立可测，既利于单元测试，也便于在作品集中单独展示。
